🧩 Structured Reasoning

Force the model to organize its thinking into clear steps and categories before responding

🤔 What Is Structured Reasoning?

Structured reasoning is a prompting technique where you instruct the model to break its internal thinking into explicit, ordered steps before producing a final answer. Instead of jumping to a conclusion, the model must show its work — like a student solving a math problem.

This improves accuracy, makes errors easier to catch, and produces outputs that are consistent and auditable.

🧰 Five Structured Reasoning Techniques

🔗 Chain-of-Thought 🔢 Numbered Steps 📁 Categorization 🏷️ XML Tags 🎭 Role Framing
  • Chain-of-Thought (CoT): Ask the model to "think step by step" or "show your reasoning" before answering.
  • Numbered Steps: Force a fixed sequence — "Step 1: identify the problem. Step 2: list options. Step 3: pick one."
  • Categorization: Require the model to sort inputs into buckets first — "Classify each item as HIGH, MEDIUM, or LOW before scoring."
  • XML Tags: Use <thinking>, <analysis>, <answer> tags to separate reasoning from output.
  • Role Framing: Assign a persona that naturally reasons — "You are a senior security auditor. Analyze this code step by step."

⚖️ Unstructured vs. Structured Prompts

Example 1: Code Review

❌ Unstructured
Review this code for bugs.
✅ Structured Reasoning
Review this code for bugs. Before
responding, organize your thinking:

1. SYNTAX — list any syntax errors
2. LOGIC — identify logic flaws
3. EDGE CASES — find unhandled cases
4. SECURITY — flag vulnerabilities
5. VERDICT — rate severity 1-5

Show each category before the final
verdict. If a category has no issues,
write "None found."

Example 2: Choosing a Tech Stack

❌ Unstructured
Should I use React or Vue for
my dashboard project?
✅ Structured Reasoning
Compare React and Vue for a
real-time dashboard project.

Before answering, think through
these categories:

1. PROJECT NEEDS — what does a
   real-time dashboard require?
2. REACT FIT — how well does
   React meet those needs?
3. VUE FIT — how well does Vue
   meet those needs?
4. TRADE-OFFS — what do you
   sacrifice with each choice?
5. RECOMMENDATION — pick one
   with a 2-sentence justification.

Example 3: Bug Diagnosis

❌ Unstructured
Why is my API returning 500?
✅ Structured Reasoning
My API returns 500 on POST /users.

Think step by step:

1. REPRODUCE — what input triggers
   the error?
2. TRACE — which middleware or
   handler is likely failing?
3. HYPOTHESES — list 3 possible
   root causes ranked by likelihood.
4. EVIDENCE — what logs or data
   would confirm each hypothesis?
5. FIX — suggest the most probable
   fix with code.

🏷️ XML Tag Separation Pattern

Claude models respond well to XML tags that separate reasoning phases. This makes it trivial to parse the final answer programmatically.

Analyze the user's request below.

<thinking>
Break the request into sub-problems.
For each sub-problem:
  - Identify what information is needed
  - Determine the approach
  - Note any assumptions
</thinking>

<analysis>
Evaluate each sub-problem's solution.
Rank them by confidence (high/med/low).
Flag any unresolved uncertainties.
</analysis>

<answer>
Provide ONLY the final answer here.
This section will be shown to the user.
Keep it concise and actionable.
</answer>
🛡️
Parsing tip: When consuming the response programmatically, extract the <answer> block only. The <thinking> block is for debugging and auditing — never show it to end users.

📁 Force Categorization Before Response

Categorization-first prompts make the model sort inputs into buckets before doing any work. This prevents hasty generalizations.

You are a triage assistant for support tickets.

For each ticket:
1. CLASSIFY the ticket type:
   - BUG (something is broken)
   - FEATURE (new capability request)
   - QUESTION (how-to or confusion)
   - BILLING (payment or account issue)

2. ASSIGN severity:
   - CRITICAL (service down, data loss)
   - HIGH (major feature broken)
   - MEDIUM (workaround exists)
   - LOW (cosmetic or nice-to-have)

3. ROUTE to the correct team based
   on your classification above.

4. RESPOND with a JSON object:
   {
     "type": "...",
     "severity": "...",
     "team": "...",
     "summary": "one sentence"
   }

Do NOT skip steps 1-3. Show your
classification before producing JSON.

🔢 Numbered Step Chains

When the task has a fixed sequence of operations, define each step explicitly and require the model to label its output by step number.

Convert this business requirement into code.

Follow these steps IN ORDER:

Step 1 — PARSE
  Extract all entities, actions, and
  constraints from the requirement.

Step 2 — MODEL
  Define the data structures needed.
  Use TypeScript interfaces.

Step 3 — IMPLEMENT
  Write the core function.
  Include error handling.

Step 4 — TEST
  Write 3 unit tests:
  - happy path
  - edge case
  - error case

Step 5 — EXPLAIN
  Provide a 3-sentence summary of
  what the code does and why.

Label each section: "Step 1 — PARSE:",
"Step 2 — MODEL:", etc.

🚫 When NOT to Use Structured Reasoning

  • Simple factual lookups: "What is the capital of France?" — forcing steps here wastes tokens.
  • Creative writing: Over-structuring kills creative flow. Use it only for the outline phase.
  • Real-time chat: Users waiting for a response don't want to see 5 reasoning steps. Use it for behind-the-scenes processing only.
  • Token-constrained calls: Structured reasoning adds 2-5x more output tokens. Budget accordingly.
⚠️
Exam trap: "Chain-of-thought always improves results" is FALSE. CoT helps with complex reasoning but adds latency and cost. For simple tasks, direct prompting is more efficient.

🔗 Combining with Other Patterns

Structured reasoning works best when combined with other prompt engineering techniques:

  • + Explicit Criteria: Define the rules the model must follow at each step (see Explicit Criteria page).
  • + Few-Shot Examples: Show one complete structured reasoning example, then let the model follow the pattern.
  • + JSON Schema: Use structured reasoning in the thinking phase, then output the final answer as validated JSON.
  • + Role Prompting: Assign a role that naturally reasons (auditor, architect, doctor) and the model will produce higher-quality steps.
