💬 Multi-Turn Prompting

Build coherent, stateful conversations that improve with each exchange

🔄 What Is Multi-Turn Prompting?

Multi-turn prompting is the practice of building a conversation across multiple user-assistant exchanges, where each turn builds on the context of previous turns. Unlike single-turn prompts, the model has access to the full conversation history.

This enables iterative refinement, progressive disclosure, and complex workflows that cannot be expressed in a single prompt.

System
You are a senior Python engineer. Respond with code and brief explanations.
Turn 1
Write a function to validate email addresses.
Turn 1
Here's a function using regex: validate_email(email) -> bool ...
Turn 2
Now add support for international domains (IDN).
Turn 2
Updated with encode('idna') for punycode conversion ...
Turn 3
Add unit tests for edge cases.
💡
Exam link: Maps to CAT05 — Context Management & Reliability (15%) and CAT04 — Prompt Engineering & Structured Output (20%). Multi-turn is the foundation of agentic workflows.

🧰 Five Multi-Turn Patterns

🔄 Refine 🌿 Branch 🧱 Build 🔧 Correct 🔬 Deepen
  • Refine: Start broad, then narrow down. "Give me 5 options" → "I like option 3, expand it" → "Add error handling to that."
  • Branch: Explore alternatives. "What if we used Redis instead?" → "Now compare that with Memcached."
  • Build: Layer complexity. "Create the schema" → "Now add the API layer" → "Now add auth."
  • Correct: Fix issues. "That has a bug on line 12" → "Now optimize the loop."
  • Deepen: Drill into specifics. "Explain the architecture" → "Zoom into the caching layer" → "What happens on cache miss?"

⚖️ Single-Turn vs. Multi-Turn

Example 1: Building an API

❌ Single-Turn (Everything at Once)
Write a REST API with Express that has
user registration, login, JWT auth,
rate limiting, input validation,
error handling, and logging. Use
PostgreSQL and include migrations.
Also add Docker and CI/CD.
✅ Multi-Turn (Progressive Build)
Turn 1: "Set up an Express project
with PostgreSQL connection and a
users table migration."

Turn 2: "Add POST /register and
POST /login endpoints with bcrypt
password hashing."

Turn 3: "Add JWT middleware that
protects all /api/* routes."

Turn 4: "Add rate limiting (100 req/
min) and input validation with Zod."

Turn 5: "Add error handling middleware
and request logging with Winston."

Turn 6: "Add Dockerfile and a GitHub
Actions CI pipeline."

Example 2: Debugging

❌ Single-Turn
My app crashes sometimes with
"Cannot read property 'id' of
undefined" but only on Tuesdays.
Here's 500 lines of code. [pastes
entire codebase]
✅ Multi-Turn (Progressive Debug)
Turn 1: "I get this error: Cannot
read property 'id' of undefined.
Here's the stack trace: ..."

Turn 2: "The error comes from
getUser(). Here's that function: ..."

Turn 3: "The user object is null
when the cache expires. Here's
the caching code: ..."

Turn 4: "Found it — the TTL is set
to 0 on Tuesdays due to a cron
job. How should I fix this?"

Example 3: Code Review

❌ Single-Turn
Review this PR. [pastes 40 files
of diff with no context about
what the PR does]
✅ Multi-Turn (Layered Review)
Turn 1: "This PR adds a payment
module. Here's the architecture
overview: [diagram]. Does the
design make sense?"

Turn 2: "Good. Now review the
Stripe integration in
src/payments/stripe.ts for
security issues."

Turn 3: "Now check the error
handling in the webhook handler.
Does it handle all Stripe event
types correctly?"

Turn 4: "Finally, review the
tests. Are edge cases covered?"

🎯 System Prompt Strategy for Multi-Turn

The system prompt sets the foundation for the entire conversation. In multi-turn, it should define:

SYSTEM PROMPT:
You are a senior backend engineer
reviewing code for production readiness.

CONVERSATION RULES:
1. Reference previous turns by saying
   "As we discussed in turn 3..."
2. When I say "go deeper", expand the
   last point with more detail.
3. When I say "next", move to the next
   topic without repeating context.
4. Maintain a running summary of
   decisions made so far.
5. If I contradict a previous decision,
   flag it explicitly.

OUTPUT FORMAT:
- Code blocks with language tags
- One-line explanation before each block
- If uncertain, state confidence level
🛡️
Pro tip: Define "control words" in your system prompt (like "go deeper", "next", "summarize"). These act as shortcuts that the model will interpret consistently across turns.

📐 Context Management Across Turns

As the conversation grows, you must manage what stays in context and what gets compressed. See the Context Compression page for details.

Turn Budget Strategy

TURNS 1-5:   Full verbatim history
             (establish context)

TURNS 6-15:  Summarize turns 1-5 into
             a 200-token block. Keep
             turns 6+ verbatim.

TURNS 16-30: Summarize turns 1-15 into
             a 300-token block. Keep
             turns 16+ verbatim.

TURN 30+:    Rolling window: keep last
             5 turns verbatim, compress
             everything else.

What to Preserve Across Turns

  • Always keep: Decisions made, file paths modified, error messages, API contracts established.
  • Compress: Long code blocks (keep signatures only), verbose explanations (keep conclusions only).
  • Drop: Greetings, acknowledgments, code that was later replaced, abandoned approaches.

🤖 Multi-Turn in Agentic Loops

Agents use multi-turn implicitly. Each tool call is a turn in a conversation between the agent and the environment.

AGENT MULTI-TURN LOOP:

Turn 1 (System):
  "You are a coding agent. Use tools
   to complete the user's task."

Turn 2 (User):
  "Add pagination to the /users endpoint."

Turn 3 (Assistant):
  [calls read_file("src/routes/users.ts")]

Turn 4 (Tool Result):
  [file contents]

Turn 5 (Assistant):
  [calls edit_file("src/routes/users.ts",
   { pagination logic })]

Turn 6 (Tool Result):
  "File updated successfully"

Turn 7 (Assistant):
  [calls run_tests()]

Turn 8 (Tool Result):
  "All 12 tests passed"

Turn 9 (Assistant):
  "Done. Added offset/limit pagination
   with ?page and ?per_page params."
⚠️
Exam trap: "Multi-turn always outperforms single-turn" is FALSE. For simple, self-contained tasks (translation, classification), single-turn is faster and cheaper. Use multi-turn only when the task requires iteration or progressive context.

🚫 Common Multi-Turn Pitfalls

  • Context drift: The model forgets earlier instructions as the conversation grows. Solution: restate key constraints in later turns.
  • Contradiction accumulation: Turn 3 says "use TypeScript", turn 10 says "switch to JavaScript". The model may hallucinate a hybrid. Solution: explicitly revoke old instructions.
  • Repetition loops: The model re-explains the same thing each turn. Solution: say "don't repeat previous context" in the system prompt.
  • Lost artifacts: Code from turn 5 gets referenced in turn 15 but was compressed out. Solution: save important code to files, not conversation history.
  • Assumption creep: The model assumes earlier context is still valid when requirements changed. Solution: use "starting fresh" to reset assumptions.
