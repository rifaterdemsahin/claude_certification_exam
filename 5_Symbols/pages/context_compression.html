🗜️ Context Compression

How to fit more meaning into fewer tokens using summary and compression techniques

📦 What Is Context Compression?

Every API call sends a context window — a fixed budget of tokens that includes the system prompt, conversation history, tool results, and the user's message. When the window fills up, older content is either truncated or the call fails.

Context compression is the practice of summarizing, deduplicating, and restructuring information so more useful knowledge fits in the same token budget.

🔧 Compression Techniques

1. Rolling Summary

After every N turns, ask the model to summarize the conversation so far, then replace the raw history with the summary.

SYSTEM PROMPT ADDITION:
After every 5 exchanges, produce a
compressed summary of the conversation.
Include:
- Decisions made
- Code changes applied
- Open questions remaining
- File paths modified

Format: structured bullet list, max
200 words. Replace prior history
with this summary.

2. Hierarchical Compression

Compress at multiple levels: full detail for the last 2 turns, bullet summary for turns 3-10, single-sentence for older.

CONTEXT LAYERS:
[LAST 2 TURNS] — full verbatim text
[TURNS 3-10]  — bullet-point summary
[OLDER]        — one sentence each
[TOOL OUTPUT]  — API signatures only

3. Selective Inclusion

Only include context that is relevant to the current query. Use a relevance filter before sending.

BEFORE EACH CALL:
1. Embed the user's new message
2. Score each context chunk for
   relevance (0.0 - 1.0)
3. Include only chunks scoring > 0.6
4. Summarize chunks scoring 0.3-0.6
5. Drop chunks scoring < 0.3

4. Structured Extraction

Instead of passing raw text, extract structured data (JSON) that captures the essential information in fewer tokens.

RAW (200 tokens):
"The user wants a React dashboard with
charts. They prefer Recharts over Chart.js
because of the API. The data comes from
a REST endpoint at /api/metrics. They want
dark theme matching their app."

EXTRACTED (60 tokens):
{
  "task": "React dashboard",
  "chart_lib": "Recharts",
  "data_source": "/api/metrics",
  "theme": "dark",
  "reason": "API preference"
}

🔢 Token Math: Why Compression Matters

Here's a real scenario showing how compression extends a conversation's useful life:

Without Compression
Context window:    200K tokens
System prompt:      2K tokens
Avg turn (user+AI): 800 tokens

200,000 - 2,000 = 198,000 available
198,000 / 800 = 247 turns max

But quality degrades after ~50 turns
because the model loses earlier context.
Effective useful turns: ~50
With Compression
Context window:    200K tokens
System prompt:      2K tokens
Compressed summary: 500 tokens (fixed)
Avg turn (user+AI): 800 tokens

After every 10 turns, compress history
into 200-token summary.

200,000 - 2,000 - 500 = 197,500
197,500 / 800 = 246 turns

+ compressed history stays at 500 tokens
so context never grows beyond budget.

Effective useful turns: 200+
⚠️
Trade-off: Compression loses nuance. A summary saying "user prefers dark theme" doesn't capture that they specifically said "match the #0f172a background." Only compress when the detail is no longer needed for future turns.

🚫 Common Exam Traps

  • "Just use a bigger model": Bigger context windows still have quadratic attention costs. Compression is always cheaper than scaling.
  • "Summarize everything": Over-compression loses critical details. Compress history, but keep the last 2-3 turns verbatim.
  • "Compression is free": The summarization call itself costs tokens and adds latency. Factor this into your budget.
  • "Drop old messages": Naive truncation breaks conversation coherence. Always replace with a summary, never just delete.
  • "One-size-fits-all": Different content needs different compression ratios. Code signatures need more fidelity than chat pleasantries.
