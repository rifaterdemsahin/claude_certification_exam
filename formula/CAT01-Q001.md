# The Three Stages of an Agentic Loop: Deep Dive

## The Core Concept

An **agentic loop** is the fundamental cycle that enables an AI agent to autonomously solve complex problems. It repeats iteratively until a goal is reached. The three stages are:

### 1. **Thinking (Planning & Reasoning)**
The agent analyzes the current state and decides what to do next.

**What happens:**
- Agent receives context (user request, current state, available tools)
- Agent reasons about the problem using its language model capabilities
- Agent selects which tool/action to use and with what parameters
- Agent plans the next step(s)

**Analogy:** *Like a chess player studying the board before making a move.* The player considers current positions, possible moves, and consequences before committing to an action.

### 2. **Action (Tool Use / Execution)**
The agent executes its decision by calling a tool, function, or API.

**What happens:**
- The selected tool is invoked with the agent's chosen parameters
- The tool performs work in the external environment (database queries, API calls, file operations, calculations)
- The action happens outside the agent's direct control—it's deterministic and grounded in reality
- Results are generated (success/failure, data returned, state changed)

**Analogy:** *Like a surgeon making an incision.* The decision is made (thinking), but now the actual cut happens in the real world with real consequences. The surgeon must now observe what actually occurred.

### 3. **Observation (Feedback & Learning)**
The agent observes the results of its action and integrates this new information into its context.

**What happens:**
- Tool results are captured and fed back to the agent
- Agent interprets what the results mean
- Agent updates its mental model of the world
- Agent decides: Is the goal achieved? Should we take another action? Do we need a different strategy?

**Analogy:** *Like checking your bank balance after a transaction.* You made the payment (action), but you need to verify it went through to know if your plan worked.

## The Loop in Action: Real Example

```
Task: "Find the average salary for software engineers in London"

┌─────────────────────────────────────────┐
│ THINK: "I need to search for salary data│
│ about software engineers in London"     │
│ Action: Call web_search tool            │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│ ACTION: Execute web_search("software    │
│ engineer salary London")                │
│ → Returns 5 relevant articles           │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│ OBSERVE: Received search results. I see │
│ mentions of £45k-£75k, but I need       │
│ specific statistics. Let me fetch one   │
│ detailed source.                        │
└──────────────┬──────────────────────────┘
               │
               ▼ (Loop repeats: THINK → ACTION → OBSERVE)
```

## Why This Matters for the Exam

### 1. **Foundational Understanding**
The agentic loop is the **conceptual backbone** of agentic systems in Claude's capabilities. Exam questions test whether you understand how agents actually work under the hood.

### 2. **Tool Use Design**
When you build Claude applications with tool use (function calling), you're orchestrating agentic loops. Understanding the stages helps you:
- Design better tools (tools are Action stage enablers)
- Anticipate failure modes (What if Action fails? How does Observe handle it?)
- Structure your prompts to encourage proper Thinking

### 3. **Error Handling & Resilience**
Exams often ask scenario-based questions like: *"Your agent's tool call failed. Where in the loop should error recovery happen?"* Answer: The **Observe stage** must detect the failure and Thinking must decide the next strategy.

### 4. **Prompt Engineering**
System prompts often explicitly guide thinking:
```
You are an agentic assistant. Follow this loop:
1. THINK: Break down the problem
2. Use tools (ACTION) only when needed
3. OBSERVE: Analyze results and decide next steps
```
Understanding this loop helps you write better instructions.

### 5. **Exam Scenarios**
Common exam patterns:
- "The agent's first tool call returned unexpected data. How should it proceed?" → Tests Observe/Replan
- "Design a tool that supports an agentic workflow" → Tests understanding of Action requirements
- "When should an agent stop looping?" → Tests understanding of terminal conditions in Observe

## Common Misconceptions (Exam Pitfalls)

❌ **Wrong:** "The agent thinks once, then acts, then observes, then stops."
✅ **Right:** "The agent **loops** through think-act-observe **multiple times** until the goal is met."

❌ **Wrong:** "Tools are optional; the agent can solve everything itself."
✅ **Right:** "The Action stage **requires** tools to ground the agent in reality; pure reasoning alone limits the agent's power."

❌ **Wrong:** "Observation is just about seeing results; it doesn't affect planning."
✅ **Right:** "Observation feeds directly into the next Thinking cycle; it's the feedback loop that drives adaptation."

## Quiz Questions

### **Question 1: Scenario-Based**

An agent is tasked with "Find me a coffee shop open after 9 PM in Cambridge and tell me their WiFi password."

The agent:
1. **Thinks:** "I need to search for coffee shops and check hours, then contact them for WiFi info"
2. **Acts:** Calls `web_search("coffee shops Cambridge open late")`
3. **Observes:** Gets 3 results; one mentions "Open until midnight"

At this point, what should the agent do next, and **which stage of the loop does this represent?**

A) Call the coffee shop's phone number immediately (Action) — it has the information it needs
B) Think about which shop to contact and what to ask them (Thinking) — this is the next cycle
C) Report the shop address and hours to the user (Observation) — we're done
D) Realize it can't get WiFi passwords and stop (Observation) — it recognizes an impossible goal

<details>
<summary>Answer</summary>
**B) Think about which shop to contact and what to ask them (Thinking)**

This is the correct interpretation. The agent has observed that the search returned partial information. The **next cycle** begins with **Thinking**: "I found a late-night shop, but I still need the WiFi password. I should contact them directly." This leads to a new **Action**: calling the shop or sending an email. The loop continues because the goal isn't yet satisfied.

**Why the others are wrong:**
- A is premature without Thinking
- C abandons the original goal (WiFi password)
- D gives up; better agents would replan
</details>

### **Question 2: Design Question**

You're building an agentic system for a bookkeeping assistant. The agent needs:
- Tool A: `query_ledger(account, date_range)` — returns transaction data
- Tool B: `calculate_balance(transactions)` — computes totals
- Tool C: `send_report(content, email)` — emails a summary

**Which tool fundamentally supports the Action stage (grounds the agent in reality), and why might the others represent a risk if called too early without proper Observation?**

A) Tool A is the only "real" tool; B and C are just processing
B) All three are equally valid tools; stage order doesn't matter
C) Tool A (query_ledger) is the only true Action; B and C should wait for results to be Observed
D) Tools don't map to stages; stages only apply to Claude's internal reasoning

<details>
<summary>Answer</summary>
**C) Tool A (query_ledger) is the only true Action; B and C should wait for results to be Observed**

**Why:**
- **Tool A (query_ledger)** is the Action stage → it queries *real data* from the ledger, grounding the agent in external reality
- **Tool B (calculate_balance)** shouldn't run until Tool A's results are Observed → the agent needs to see what data was returned before processing it
- **Tool C (send_report)** is even later → it's part of a *subsequent* Observe-Think cycle where the agent realizes "I have the data, now I should report it"

Calling C without Observing A would mean emailing stale or incorrect data.

**Why the others are wrong:**
- A oversimplifies; B and C are useful but dependent on sequencing
- B ignores the feedback loop's importance
- D is false; stages absolutely map to tool sequences in well-designed agents
</details>

## Study Tips for the Exam

1. **Draw the loop:** Sketch it out on exam day. It grounds your thinking.
2. **Think in cycles:** Ask "what does the agent know NOW? What's the next loop?" 
3. **Anticipate tool failures:** Every tool call in Action can fail. Where does that get handled? (Observation)
4. **Remember the purpose:** The loop enables **iterative refinement**—each cycle gets closer to the goal.
5. **Connect to prompting:** Your system prompts should encourage explicit Thinking, appropriate Action, and careful Observation.

Good luck! 🚀
