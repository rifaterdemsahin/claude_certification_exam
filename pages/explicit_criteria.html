📋 Explicit Criteria & Constraints

How to define unambiguous requirements that model outputs must satisfy

🎯 What Are Explicit Criteria?

Explicit criteria are precise, testable rules that dictate what a model must or must not produce. They remove ambiguity, reduce hallucination, and make output validation possible.

Implicit instructions rely on the model's general training (e.g., "be helpful"). Explicit instructions leave no room for interpretation (e.g., "respond in exactly 3 bullet points, each under 10 words").

🧰 The Five Types of Constraints

Every explicit criterion falls into one of these categories. Tag your prompts with them to ensure coverage:

📐 Format 📏 Length 🔢 Range 📎 Enum 🚫 Forbidden
  • Format: JSON, Markdown table, bullet list, XML, YAML, regex pattern.
  • Length: Max tokens, max words, exact character count, min/max paragraphs.
  • Range: Numeric bounds (1–5), date windows, confidence thresholds.
  • Enum: Closed set of allowed values ("only: HIGH, MEDIUM, LOW").
  • Forbidden: Prohibited words, topics, assumptions, or output patterns.

⚖️ Implicit vs. Explicit Prompts

Example 1: Summarization

❌ Implicit / Vague
Summarize this article.
✅ Explicit / Constrained
Summarize the article in exactly 3 bullet points.
Each bullet must be under 15 words.
Include only facts mentioned in the text.
Do not add interpretation or opinion.
Output as a JSON array of strings.

Example 2: Sentiment Analysis

❌ Implicit / Vague
What is the sentiment of this review?
✅ Explicit / Constrained
Classify the review sentiment.
Allowed values: POSITIVE, NEUTRAL, NEGATIVE.
Provide a confidence score 0.0–1.0.
If mixed, pick the dominant sentiment.
Respond ONLY with valid JSON: {"sentiment": "...", "confidence": 0.0}

Example 3: Code Generation

❌ Implicit / Vague
Write a Python function to sort users.
✅ Explicit / Constrained
Write a Python function `sort_users(users)`.
Input: list of dicts with keys "name" (str) and "age" (int).
Sort by age ascending, then name ascending.
Return a new list; do not mutate the input.
Use only the standard library (no external packages).
Include a 3-line docstring and type hints.
Max function length: 20 lines.

📐 Structured Output with JSON Schema

When you need machine-parseable results, bind the model to a strict schema. This turns soft constraints into hard validation rules.

{
  "name": "risk_assessment",
  "schema": {
    "type": "object",
    "properties": {
      "risk_level": {
        "type": "string",
        "enum": ["LOW", "MEDIUM", "HIGH", "CRITICAL"]
      },
      "score": {
        "type": "integer",
        "minimum": 0,
        "maximum": 100
      },
      "mitigations": {
        "type": "array",
        "items": { "type": "string" },
        "minItems": 1,
        "maxItems": 5
      },
      "flagged": { "type": "boolean" }
    },
    "required": ["risk_level", "score", "mitigations", "flagged"],
    "additionalProperties": false
  }
}
🛡️
Validation tip: Always set additionalProperties: false to prevent the model from inventing keys outside your schema. This is a common exam trap.

✅ Explicit Criteria Checklist

Before sending a prompt to production, verify every item below:

✓ The output format is named (JSON, Markdown, CSV, plain text, etc.).
✓ Length limits are specified (max tokens, words, characters, items).
✓ Allowed values are listed when the output is categorical (enum).
✓ Numeric fields have min/max bounds or precision rules.
✓ Forbidden content is explicitly blocked (topics, tone, assumptions).
✓ Required vs. optional fields are distinguished.
✓ Examples of "good" output are provided in the prompt (few-shot).
✓ A validation step is described ("check that X before responding").

🚫 Common Exam Traps

  • Assuming implicit knowledge: "Use best practices" is not a criterion. Name the practice.
  • Missing edge cases: If the input is empty, the model must still follow the schema.
  • Over-constraining: Too many rigid rules can lower quality. Balance specificity with room for reasoning.
  • No validation hook: Always tell the model how to verify its own output before returning it.
  • Conflicting constraints: "Respond in 1 sentence" + "include all 10 data points" is impossible.
